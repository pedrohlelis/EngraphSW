## Purpose

This document defines the long-term brand identity, visual philosophy, and experiential direction of EngraphSW.

These guidelines exist to maintain consistency across:
- UI design
- UX decisions
- documentation
- AI-generated content
- marketing
- onboarding
- architecture communication

---

# Product Identity

EngraphSW is an AI-native graph workspace for software engineering.

The platform enables teams to model software engineering knowledge, traceability, dependencies, estimation, and SDLC artifacts as interconnected graph structures.

The platform is:
- graph-native
- engineering-focused
- AI-assisted
- traceability-centric
- architecture-oriented

The platform is NOT:
- a generic project management tool
- a Kanban board
- a no-code automation platform
- a consumer productivity app
- a BPMN workflow system

---

# Brand Personality

EngraphSW should feel:

- technical
- intelligent
- professional
- precise
- calm
- structured
- modern
- engineering-oriented

The platform should communicate:
- trust
- clarity
- control
- engineering rigor
- system intelligence

Avoid:
- hype-driven aesthetics
- excessive gamification
- consumer-app playfulness
- startup cliché visuals
- marketing-heavy UX patterns

---

# Visual Identity Philosophy

The interface should resemble:
- professional engineering tools
- graph-native developer systems
- modern software architecture platforms

Inspirational references:
- Claude Code
- Linear
- Raycast
- Vercel
- modern IDE ecosystems

---

# Design Principles

Prioritize:
- graph readability
- information clarity
- structured layouts
- visual restraint
- execution visibility
- dependency comprehension

Avoid:
- visual clutter
- oversized UI chrome
- unnecessary animations
- decorative gradients
- excessive whitespace

---

# Theme Direction

Use:
- dark-first interface
- layered dark surfaces
- restrained color palette
- subtle depth
- strong typography hierarchy
- compact controls

The interface should feel:
- focused
- immersive
- system-oriented

---

# Color Philosophy

Primary accent:
- warm amber/orange

Accent usage:
- active focus
- node selection
- graph highlights
- execution states
- dependency emphasis

Semantic colors:
- success -> muted green
- warning -> amber
- error -> muted red
- informational -> cool muted blue

Avoid:
- rainbow dashboards
- overly saturated colors
- excessive semantic noise

---

# Surface Philosophy

Preferred palette direction:
- near-black backgrounds
- charcoal surfaces
- muted gray borders
- warm light typography
- subtle graph glow accents

---

# Typography Philosophy

Typography should prioritize:
- readability
- density
- hierarchy
- graph scanning efficiency

Prefer:
- compact spacing
- strong contrast
- consistent sizing
- restrained font scaling

---

# UX Philosophy

The platform should feel like:
- a software engineering operating system
- an engineering knowledge graph
- an intelligent CASE environment
- a graph-native SDLC workspace

The user should feel like they are modeling engineering systems, not managing tasks.

---

# Node Design Philosophy

Nodes are engineering entities first and visual components second.

Node interfaces should emphasize:
- structure
- traceability
- execution visibility
- dependency awareness
- engineering context

Avoid:
- decorative card UI
- generic dashboard widgets
- consumer productivity aesthetics

---

# AI Philosophy

AI should behave as:
- an engineering assistant
- a reasoning augmentation layer
- a system intelligence capability

AI should never feel:
- magical
- opaque
- autonomous
- uncontrollable

Users should always feel:
- informed
- in control
- capable of reviewing outputs
