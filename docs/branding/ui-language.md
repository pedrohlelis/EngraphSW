# EngraphSW — UI Language Guidelines

## Purpose

This document defines the communication style, terminology usage, UX writing philosophy, and interface language standards for EngraphSW.

The platform should communicate with:
- precision
- clarity
- engineering-oriented structure
- calm professionalism

---

# UX Writing Philosophy

The interface should feel:
- technical
- structured
- informative
- trustworthy
- engineering-oriented

Users should feel:
- in control
- informed
- capable of reasoning about the system

Avoid:
- excessive friendliness
- motivational SaaS language
- hype-driven copy
- playful UX writing

---

# Preferred Tone

Preferred tone:
- concise
- professional
- calm
- direct
- analytical

The interface should sound like:
- an advanced engineering tool
- a professional developer platform
- an intelligent modeling environment

---

# Avoid Marketing Language

Avoid:
- revolutionize your workflow
- supercharge productivity
- magical AI
- game-changing platform
- next-generation synergy

Prefer:
- structured
- measurable
- traceable
- analyzable
- explainable

---

# Preferred Terminology

| Preferred | Avoid |
|---|---|
| Artifact | Card |
| Node | Block |
| Traceability | Link |
| Dependency | Automation |
| Execution | Workflow Run |
| Workspace | Board |
| Propagation | Sync Magic |

---

# Button Language

Prefer:
- Generate Requirements
- Analyze Traceability
- Propagate Dependencies
- Recalculate Metrics
- Validate Consistency

Avoid:
- Boost Workflow
- Smart Generate
- One-Click Magic
- Quick Win

---

# Empty State Philosophy

Empty states should:
- explain purpose
- guide next action
- remain concise

Preferred:
- “Create a Functional Requirements Node to begin modeling system behavior.”

Avoid:
- “Looks lonely here”
- “Let’s get productive!”

---

# Error Messaging Philosophy

Errors should explain:
- what failed
- why it failed
- affected dependencies
- possible resolution paths

Preferred:
- “Dependency propagation failed because the upstream node schema changed.”

Avoid:
- “Something went wrong.”

---

# AI Communication Rules

AI-generated outputs should:
- remain reviewable
- remain editable
- remain explainable

The interface should present AI as:
- assistance
- augmentation
- engineering support

NOT:
- autonomous authority
- unquestionable truth
- black-box intelligence

Preferred:
- “AI-generated suggestion”
- “AI-assisted analysis”
- “Review generated requirements”

Avoid:
- “The AI decided”
- “Automatically solved”
- “Perfect solution generated”

---

# Graph Interaction Language

Prefer:
- Connect Nodes
- Trace Dependencies
- Analyze Relationships
- Visualize Propagation

Avoid:
- Build Automation
- Create Pipeline
- Run Zap

---

# Notification Philosophy

Notifications should:
- be minimal
- informative
- non-disruptive

Avoid:
- excessive celebration
- emoji-heavy feedback
- gamified interaction patterns

---

# Documentation Language

Documentation should:
- prioritize precision
- explain reasoning
- describe architecture clearly
- maintain engineering-oriented terminology

Avoid:
- vague abstraction
- unnecessary buzzwords
- oversimplified technical explanations
